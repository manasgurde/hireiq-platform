# External Integrations

**Analysis Date:** 2026-06-17

## APIs & External Services

**Internal Microservices:**
- AI / ML Service - Specialized FastAPI service hosting spaCy and sentence-transformer models
  - Integration Method: Async HTTP REST calls from Core API via `httpx` client (`backend/app/services/ai_service.py`)
  - Endpoints: `POST /ai/analyze-resume`, `POST /ai/score-candidate`, `POST /ai/recommend-jobs`, `POST /ai/evaluate-interview`, `GET /health`
  - URL: Configured via `AI_SERVICE_URL` environment variable (e.g., `http://ai:8001` in local compose, internal VPC DNS in production)

**Email/SMS (Planned v1.1):**
- Transactional email service for candidate status notifications (applied, shortlisted, rejected)
  - Integration: SMTP or HTTP client SDK (e.g., SendGrid/SES)

## Data Storage

**Databases:**
- AWS RDS PostgreSQL 15 - Primary persistent datastore
  - Connection: Connection pool handled via SQLAlchemy async engine using `DATABASE_URL` env var
  - Client: `SQLAlchemy 2.0` with `asyncpg` driver in `backend/app/core/database.py`
  - Migrations: Managed via `Alembic` under `backend/migrations/`

**File Storage:**
- AWS S3 - Private object storage for resumes
  - SDK/Client: Python `boto3` SDK in `backend/app/utils/s3.py`
  - Auth: IAM role credentials (production) or `AWS_ACCESS_KEY_ID` / `AWS_SECRET_ACCESS_KEY` environment variables
  - Bucket: Configured via `S3_BUCKET_NAME` env var (e.g., `hireiq-resumes-prod`)
  - Flow: Resume files uploaded directly via 15-minute expiry pre-signed URLs generated by `generate_upload_url()`

**Caching & Key-Value Store:**
- AWS ElastiCache Redis 7 - Distributed cache, rate limiter, and refresh token store
  - Connection: connection pool via `redis-py` client in `backend/app/core/redis.py`
  - Auth: Redis AUTH password connection URL in `REDIS_URL` env var
  - Patterns: job listings caching (5 min), user profiles (10 min), AI scores (1 hour), rate limiting sliding windows, and refresh tokens

## Authentication & Identity

**Auth Provider:**
- Custom JWT-based stateless authentication with dual-token strategy
  - Access Token: JWT signed using HMAC-SHA256 (24-hour expiry)
  - Refresh Token: Hashed UUID stored in Redis (30-day expiry)
  - Token Storage: Access token in client Zustand memory (never localStorage), refresh token in httpOnly secure cookie
  - Endpoint Routing: JWT validation middleware intercepting routes (`backend/app/middlewares/auth_middleware.py`)

## Monitoring & Observability

**Logs:**
- AWS CloudWatch + Datadog - Metrics, alerts, and log aggregation
  - Integration: `structlog` in Python writing structured JSON to stdout, shipped via CloudWatch Agent

**Alerting:**
- PagerDuty - Connected to CloudWatch alerts for critical triggers (5+ failed logins, 100+ req/min, S3 policy changes, high DB CPU)

## CI/CD & Deployment

**Hosting:**
- AWS ECS Fargate - Containerized serverless compute hosting frontend and backend tasks
- AWS CloudFront - Static CDN serving Next.js assets and proxying S3 files

**CI Pipeline:**
- GitHub Actions - Build and deployment pipeline (`.github/workflows/ci.yml`)
  - Secrets: Docker registry credentials, AWS deployment role keys
  - Workflows: Lint (ruff + eslint) -> Test (pytest + jest) -> Build -> Push (ECR) -> Deploy (ECS Fargate rolling) -> Smoke Tests -> Slack Notification

## Environment Configuration

**Development:**
- Required env vars: `DATABASE_URL`, `REDIS_URL`, `AI_SERVICE_URL`, `AWS_ACCESS_KEY_ID`, `AWS_SECRET_ACCESS_KEY`, `S3_BUCKET_NAME`, `JWT_SECRET_KEY`
- Secrets location: `.env` files (gitignored)
- Mock services: Local PostgreSQL & Redis containers, mocks used for S3 in local tests

**Production:**
- Secrets management: AWS Secrets Manager injecting credentials at container runtime
- Security: All service communications locked behind private subnets (NAT Gateway), TLS 1.3 termination at Load Balancer
