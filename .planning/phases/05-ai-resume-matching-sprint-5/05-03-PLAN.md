# Phase 5, Plan 3: AI Service Client (HIQ-050)

**Target Phase:** 05-ai-resume-matching-sprint-5
**Goal:** Implement the `AIService` HTTP client in the Core API with timeouts, Redis score caching, and silent error logging fallbacks to prevent the web system from going down if the ML service crashes.

## Technical Context
- **Dependencies:** 05-RESEARCH.md (Core API Client with Fallbacks & Redis Caching)
- **New files:** `backend/app/core/ai_client.py`
- **Modify:** `backend/app/api/v1/applications.py`, `backend/app/api/v1/resumes.py` (if triggering syncs)

## Action Plan

### 1. `AIServiceClient` Core Utility (`backend/app/core/ai_client.py`)
- [ ] Create `AIServiceClient` class initialized with the `AI_SERVICE_URL` (e.g. `http://localhost:8001`) and the `redis` client.
- [ ] Implement `async def get_matching_score(self, resume_id, job_id, resume_text, job_text) -> float`:
  - Check Redis for `score:matching:{resume_id}:{job_id}`.
  - Return if cached.
  - If not cached, `httpx.AsyncClient` `POST /match` with `timeout=10.0`.
  - Cache result in Redis for 24 hours.
  - On `TimeoutException` or `RequestError`, silently log to `structlog` and return `0.0`.
- [ ] Implement `async def extract_resume_entities(self, text: str) -> dict`:
  - `POST /parse` with fallback to empty dicts `{}` on failure.
- [ ] Implement `async def evaluate_interview(self, cand_ans: str, ideal_ans: str) -> float`:
  - `POST /evaluate-interview` with fallback `0.0`.

### 2. Integration
- [ ] Ensure `httpx` is in `backend/requirements.txt` (should be natively available via `fastapi`/`starlette` testing dependencies, but add explicitly to be safe).
- [ ] Inject the `AIServiceClient` into the application lifecycle or as a FastAPI `Depends()`.

## Verification
- [ ] Unit tests or manual curl calls to `AIServiceClient` correctly fallback to default values if port 8001 is closed.
- [ ] Successful calls are cached in Redis and bypass the ML container on the second request.
